📖 AI Assistant Guide: Docker Network Configuration
Here is the step-by-step manual for configuring the docker-compose.yml file to ensure all containers can communicate correctly.

Objective
To create a single, unified Docker network where all 13 services can discover and communicate with each other by their service name. This guide also explains how to expose specific services to the host machine for external access (e.g., API testing, viewing a UI).

Step 1: Define a Custom Network
First, instead of relying on the default network, we'll create a custom bridge network. This is best practice as it provides better isolation and allows containers to resolve each other by name.

Add this block at the very end of your docker-compose.yml file:

YAML

networks:
  apexsigma_net:
    driver: bridge
Step 2: Assign All Services to the Custom Network
For every single service defined in your docker-compose.yml file, you must tell it to connect to the apexsigma_net network.

Here is an example for the memos-api service. Notice the networks block at the end.

YAML

services:
  memos-api:
    image: memos-api-image:latest
    container_name: memos-api
    # ... other configurations like environment variables ...
    networks:
      - apexsigma_net
Instruction: Apply this networks block to all 13 services (PostgreSQL, Redis, RabbitMQ, Grafana, etc.).

Step 3: How Containers Talk to Each Other (Internal Communication)
Once all services are on the same custom network, they can communicate using their service name as the hostname.

Example:
The memos-api service needs to connect to its PostgreSQL database. The database service is named postgres-memos in the docker-compose.yml file.

In the memos-api application's configuration (e.g., an environment variable), the database host should be set to postgres-memos, not localhost or an IP address.

Service Name: postgres-memos

Database Port (internal): 5432

Connection String: postgresql://user:password@postgres-memos:5432/memos_db

Docker's internal DNS will automatically resolve postgres-memos to the correct container's internal IP address on the apexsigma_net network. This works for any service talking to any other service.

Step 4: How to Connect from the Outside (External Access)
To access a container from your host machine (e.g., using your web browser or an API client like Postman), you must map a port from the container to your host machine. This is done using the ports directive.

The format is "HOST_PORT:CONTAINER_PORT".

Example:
The memos-api service runs on port 8000 inside its container. To access it from your computer, you map it to a port on your host machine, like 8001.

YAML

services:
  memos-api:
    # ... other configurations ...
    ports:
      - "8001:8000" # Exposes container port 8000 on host port 8001
    networks:
      - apexsigma_net
Instruction: Now, to send a request to the API from your machine, you would use http://localhost:8001.

Apply this logic to any service that needs external access:

Grafana: map to 3000:3000 to access the UI at http://localhost:3000.

Langfuse: map to 3300:3000 to access its UI at http://localhost:3300.

PostgreSQL (for debugging): You could map 5433:5432 to connect with a database tool, but this is often not needed and less secure.

By following these steps, you will have a robust and correctly configured network for the entire ecosystem. ✨
